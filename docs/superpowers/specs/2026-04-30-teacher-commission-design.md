# Teacher Commission Design

**Date:** 2026-04-30  
**Status:** Approved

---

## Overview

Track teacher earnings as a percentage of each payment allocated to their lessons. Show potential earnings (conducted but unpaid) separately from official earnings (payment-backed). Support manual payouts with full history.

---

## Section 1: Data Model

### TeacherCommission
Stores commission rate history per teacher. Never overwritten — each rate change creates a new record.

| Field | Type | Notes |
|-------|------|-------|
| id | String (cuid) | PK |
| teacherId | String | FK → User |
| percentage | Decimal | e.g. 70.00 |
| effectiveFrom | DateTime | Rate applies from this date |
| createdById | String | FK → User (admin who set it) |
| createdAt | DateTime | |

### TeacherEarning
One record per PaymentLesson where the corresponding lesson is CONDUCTED.

| Field | Type | Notes |
|-------|------|-------|
| id | String (cuid) | PK |
| teacherId | String | FK → User |
| lessonId | String | FK → Lesson |
| paymentLessonId | String | FK → PaymentLesson (cascade delete) |
| amount | Decimal | Computed: pl.amount × (pct / 100) |
| percentage | Decimal | Snapshot of rate at lesson.startDate |
| createdAt | DateTime | |

Cascade delete on `paymentLessonId` ensures TeacherEarning is cleaned up automatically when `reallocate` wipes PaymentLesson records.

### TeacherPayout
Manual payout recorded by admin.

| Field | Type | Notes |
|-------|------|-------|
| id | String (cuid) | PK |
| teacherId | String | FK → User |
| amount | Decimal | Amount paid out |
| adminId | String | FK → User (who recorded it) |
| notes | String? | Optional |
| createdAt | DateTime | |

### SchoolTransaction (extension)
Add new reason enum value: `LESSON_SCHOOL_SHARE`.  
Add optional fields: `lessonId String?` and make `adminId` nullable.

## Section 2: Backend Logic

### reallocate extension

`reallocate(childId, teacherId)` is extended to recalculate TeacherEarning after redistributing PaymentLesson records:

1. Delete all `PaymentLesson` for the pair → cascades to `TeacherEarning`
2. Delete all `SchoolTransaction(LESSON_SCHOOL_SHARE)` where `lessonId` belongs to this pair
3. Run `computeAllocation` → create new `PaymentLesson` records
4. For each new `PaymentLesson` where the lesson status is `CONDUCTED`:
   - Find active `TeacherCommission`: `WHERE teacherId = ? AND effectiveFrom <= lesson.startDate ORDER BY effectiveFrom DESC LIMIT 1`
   - If commission found:
     - Create `TeacherEarning(amount = pl.amount × pct/100, percentage = pct)`
     - Create `SchoolTransaction(LESSON_SCHOOL_SHARE, amount = pl.amount × (1 - pct/100), lessonId)`

### Teacher Balance Endpoint

`GET /teachers/:id/balance` returns:

```typescript
{
  officialEarnings: number,   // Σ TeacherEarning
  potentialEarnings: number,  // CONDUCTED lessons with no PaymentLesson × current pct
  totalPayout: number,        // Σ TeacherPayout
  balance: number,            // officialEarnings - totalPayout
  potentialBalance: number,   // (officialEarnings + potentialEarnings) - totalPayout
  currentCommission: number | null,
}
```

Potential earnings: lessons with status `CONDUCTED` and no `PaymentLesson` records, valued at `lesson.price × latestCommission.percentage / 100`.

### Commission Management

`POST /teacher-commissions` — create new commission record (never update existing).  
`GET /teacher-commissions?teacherId=` — list history for a teacher.

### Payout Management

`POST /teacher-payouts` — record payout, create `SchoolTransaction(TEACHER_PAYOUT)`.  
`GET /teacher-payouts?teacherId=` — list payout history.

## Section 3: Frontend

### New page: `/commissions` (admin only)

**Teacher list view** — table or cards per teacher:
- Avatar + name
- Official balance (green): `officialEarnings - totalPayout`
- Potential balance (grey/italic): includes unconfirmed conducted lessons
- "Виплатити" button → payout modal (amount + notes)

**Teacher detail** (expand or click):
- Current commission rate + "Змінити ставку" button → inline form (percentage + effective date)
- Commission history (collapsible)
- Earnings table: one row per TeacherEarning — date, student, payment amount, commission amount
- Potential earnings section (grey): conducted unpaid lessons
- Payout history

**Commission rate form:**
- Percentage input
- Effective-from date (defaults to today)
- Saves as new TeacherCommission record

### Out of scope (this iteration)
- Export / reports
- Bulk payouts
- Per-lesson commission overrides

---

## Key Invariants

- TeacherEarning records are always derived — never manually created; always regenerated by `reallocate`
- Commission rate used is the one effective on `lesson.startDate`, not payment date
- If no commission is configured for a teacher, no TeacherEarning or SchoolTransaction is created (silently skipped)
- Cascade delete on PaymentLesson ensures no orphaned TeacherEarning records
